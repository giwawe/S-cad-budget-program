# GUI v1 需求边界

## 目标

GUI v1 是面向预算员和设计师的本地桌面操作台。它把已经稳定的 CAD 导入、算量、报价、单价表和真实验收流程包装成可点击流程，减少命令行操作和路径输入错误。

第一版不重新实现算量或报价逻辑；所有业务结果仍来自现有 Python 核心和脚本。

## 技术选型结论

建议第一版使用 `PySide6`。

理由：

- 目标用户是预算员和设计师，桌面窗口、文件选择器、按钮、表格和打开目录更符合使用习惯。
- 项目核心已经是 Python，GUI 可以直接调用现有服务函数，不需要先引入前后端分离架构。
- `PySide6` 后续可以承载表格、日志、进度条、多页面导航和 QSS 样式，比命令行 TUI 更适合交付给非开发用户。

备选：

- `Textual`：适合开发者内部工具，开发快，但对非开发用户不如桌面 GUI 直观。
- 本地 Web 应用：适合远程访问或多人协作，但第一版会增加服务、端口、浏览器和部署复杂度。

## v1 必须做

- 选择 DXF 文件。
- 选择商品房报价模板 Excel。
- 选择全局单价表 Excel。
- 选择输出目录。
- 一键生成正式报价包。
- 一键运行真实验收。
- 显示核心统计：
  - 自动算量
  - 自动汇总
  - 模板默认
  - 自动生成-默认推断
  - 自动生成-异常提示
  - high / medium / low 复核行动数量
  - 单价匹配行数
- 显示复核行动列表，至少包含行动名称、优先级、影响报价行数、涉及项目。
- 显示输出文件路径，并提供打开输出目录按钮。
- 运行失败时显示清晰错误，不把 Python traceback 作为主要信息。

## v1 不做

- 不做 CAD 编辑器。
- 不做 CAD 图纸可视化绘图或图层在线修改。
- 不做报价 Excel 内嵌编辑器。
- 不做主材库、客户库、方案历史管理。
- 不做多人协作、登录、权限、云端同步。
- 不把全量 `pytest` 放进 GUI 的普通业务按钮；全量测试仍是开发流程。
- 不让交付后的 GUI 依赖 Codex skill、联网 AI 或当前开发环境。

## 页面结构

### 1. 项目运行页

用于日常生成报价。

主要控件：

- DXF 文件选择。
- 模板 Excel 选择。
- 单价表 Excel 选择。
- 输出目录选择。
- “生成正式报价”按钮。
- “运行真实验收”按钮。
- 状态区：空闲、运行中、成功、失败。
- 日志区：显示步骤摘要和错误信息。

### 2. 结果摘要页

用于查看最近一次运行结果。

主要控件：

- 自动化统计卡片。
- 复核状态统计卡片。
- 行动优先级统计卡片。
- 复核行动表。
- 输出文件列表。
- 打开输出目录按钮。

### 3. 设置页

用于保存默认路径。

主要控件：

- 默认模板路径。
- 默认单价表路径。
- 默认输出根目录。
- DXF 默认单位，第一版默认 `mm`。

## 成功标准

- 预算员无需命令行即可生成 `quote-priced.xlsx` 和复核包。
- 设计师能看到 high / medium 复核行动，并知道哪些需要补 CAD。
- GUI 输出的结果与 `scripts\run_real_acceptance.py` 一致。
- 任意 GUI 运行失败时，用户能看懂失败发生在“输入检查 / CAD 导入 / 算量 / 报价 / 单价校验 / 输出校验”哪一步。

## 进入实现前的约束

- GUI 不直接解析 Excel 或 DXF 的业务规则，必须调用现有 Python 服务函数。
- GUI 不直接拼长命令行作为主要实现方式；命令行脚本只作为人工 fallback。
- GUI 第一版使用本地文件路径，不引入数据库。
- 真实单价表仍不提交仓库。
